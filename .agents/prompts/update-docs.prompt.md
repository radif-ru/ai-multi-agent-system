# Промпт: обновить документацию

Действуй как инженер, поддерживающий документацию проекта в `_docs/` и `_board/`. Я описываю изменение поведения (или указываю расхождение между кодом и документом).

Принципы (`_docs/instructions.md` §9):

- **Источник истины — код.** При расхождении документ приводится в соответствие коду отдельным `docs(...)`-коммитом, а не наоборот.
- **Только относительные пути.** Запрещены абсолютные пути ФС (`/home/...`, `C:\...`). Все ссылки — относительно корня (`_docs/...`, `_board/...`, `app/...`).
- **Синхронизация ссылок.** При переименовании/переносе/изменении номера раздела — пройти `grep -rn` по репо и поправить все живые ссылки. Закрытые спринты (`_board/sprints/<NN>-*.md` со статусом `Closed`) — не трогать (это архив).
- **Язык.** Текст — на русском; технические идентификаторы (модули, функции, env, пути) — латиницей.

Куда что пишем:

- **`_docs/`** — *что и как устроено*: требования, архитектура, протокол цикла, форматы tools/skills/prompts, инструкции, тестирование, безопасность.
- **`_board/`** — *что сейчас делаем*: активный спринт, его задачи, процесс, история.
- **`README.md`** — обновлять при новой команде / параметре / зависимости.

Выбери правильный документ по теме изменения (`tools.md`, `skills.md`, `commands.md`, `memory.md`, `agent-loop.md`, `security.md`, `observability.md`, `multi-agent.md` и др.). Не дублируй одно и то же в нескольких местах — ссылайся. Перед завершением перечисли, какие файлы изменены и какие ссылки проверены через `grep`.
