# Промпт: диагностика и исправление бага

Действуй как инженер, который чинит баги через **корневую причину**, а не симптомы. Я описываю проблему (и/или прикладываю лог/стектрейс).

Порядок работы:

1. **Сформулируй гипотезу.** Что именно сломано и почему — на основе кода и логов, без догадок. Если данных не хватает, перечисли, какие логи/входные данные нужны.
2. **Воспроизведи багом-тестом.** Напиши падающий unit-тест, который воспроизводит дефект (по правилам `_docs/testing.md`: моки внешних систем, без сети). Это критерий успеха: тест красный → после фикса зелёный.
3. **Найди корневую причину.** Покажи конкретное место (файл, строка, условие). Отличай причину от симптома.
4. **Исправь минимально.** Предпочитай однострочную/точечную правку у источника проблемы, а не обходной костыль ниже по потоку. Не трогай несвязанный код (`AGENTS.md` §3).
5. **Проверь.** Баг-тест зелёный, остальной `pytest -q` зелёный, `flake8` чистый на изменённых файлах.

Если при отладке нужно больше наблюдаемости — добавляй точечные `logger.debug`/`logger.info` с осмысленными полями, а не `print`. Сообщения логов — на русском, структурные поля и идентификаторы — латиницей.

Не маскируй падение через `pytest -k`/`skip` и не ослабляй существующие тесты.
