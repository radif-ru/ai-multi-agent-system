# Промпт: добавить новый skill

Действуй как инженер этого проекта. Я описываю **класс задач**, для которого нужен skill — markdown-инструкция, которую агент подгружает в цикле через tool `load_skill`. Реализуй по формату `_docs/skills.md`.

Сначала проверь: это точно skill, а не tool? Skill — это **знание о том, как решать класс задач** (последовательность шагов, формат ответа, эвристики), оформленное markdown'ом. Если нужна **программа** (вычисление, сеть, чтение файла) — это tool (`_docs/skills.md` §1).

Формат (`_docs/skills.md` §2–§3):

- Каталог `app/skills/<name>/SKILL.md`. Имя подпапки = имя скилла для `load_skill`, в kebab-case.
- Описание — либо первая строка `Description: <одна фраза ≤ 200 символов>`, либо YAML frontmatter с полем `description`. Описание видит **каждый** агентный цикл, поэтому оно короткое и сразу показывает триггер срабатывания.
- Тело скилла включает обязательные разделы: заголовок `# Skill: <name>`, «Когда использовать» (критерии срабатывания), «Алгоритм» (пошаговая инструкция).
- Желательно: «Когда **не** использовать», 1–2 примера (вход → выход), «Чего избегать» (типичные галлюцинации).

Что НЕ кладём в skill (`_docs/skills.md` §7): код (`.py`/shell), системный промпт, описания tools, справочную документацию для человека.

Шаги (`_docs/skills.md` §5):

1. Создать `app/skills/<name>/SKILL.md` по формату выше.
2. Перезапустить процесс — `SkillRegistry` подхватит скилл на старте (hot-reload не поддерживается).
3. Smoke-проверка: задать боту задачу, для которой нужен скилл; убедиться, что в логах есть `tool=load_skill name=<name>`.

Тесты на сам контент скилла не пишем (это контент, а не код); тесты на `SkillRegistry` — общие.
